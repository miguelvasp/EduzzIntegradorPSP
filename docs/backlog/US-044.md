# US-044 — Fechar o contrato HTTP e tratamento global de erro

## 1. Identificação

- **ID:** US-044
- **Título:** Fechar o contrato HTTP e tratamento global de erro
- **Épico:** API de consulta e exposição funcional
- **Tipo:** Backend / API / Governança de contrato
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-011 — Padronizar tratamento de erros da aplicação; US-023 — Implementar endpoint de listagem paginada de transações; US-024 — Implementar endpoint de detalhe de transação; US-025 — Implementar endpoint de listagem de parcelas da transação; US-026 — Implementar endpoint de detalhe de parcela; US-027 — Implementar endpoint de consulta de pagador da transação; US-028 — Implementar contrato padronizado de resposta e paginação; US-042 — Implementar mecanismo acionável de sincronização de verdade; US-043 — Implementar os endpoints REST obrigatórios com consulta real

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** fechar o contrato HTTP e o tratamento global de erro da aplicação  
**para** garantir consistência real entre endpoints, reduzir ambiguidade para consumidores e avaliadores, e impedir que erros internos vazem de forma desorganizada ou insegura.

---

## 3. Contexto de negócio

Uma API não é avaliada só pelo caminho feliz. Ela é avaliada pelo comportamento inteiro.

Se cada endpoint responde com um formato diferente, se erro de validação vira 500, se recurso inexistente vira mensagem solta, ou se stack trace vaza para fora, a entrega perde credibilidade técnica muito rápido.

O desafio exige endpoints REST funcionais e respostas de erro consistentes. Isso significa que o contrato HTTP não pode estar meio certo. Tem que estar fechado.

Essa User Story existe para eliminar os buracos mais comuns e mais feios:

- resposta inconsistente entre endpoints;
- paginação com shape diferente do contrato padrão;
- 400, 404, 422 e 500 mal tratados;
- erro interno vazando detalhe desnecessário;
- ausência de `requestId` ou identificador de correlação útil;
- validação espalhada e imprevisível.

Sem isso, a API até responde, mas responde mal.

---

## 4. Objetivo

Fechar o contrato HTTP e o tratamento global de erro, contemplando:

- padronização definitiva das respostas de sucesso;
- padronização definitiva das respostas de erro;
- mapeamento consistente de falhas para status HTTP corretos;
- correlação mínima por `requestId` ou identificador equivalente;
- base estável para Swagger, E2E, troubleshooting e avaliação técnica.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- consolidação do contrato HTTP do projeto;
- ajuste dos endpoints obrigatórios ao contrato único;
- implementação ou fechamento do middleware global de erro;
- mapeamento consistente de exceções e erros de domínio;
- padronização de respostas de validação e não encontrado;
- inclusão de identificador de correlação nas respostas de erro quando aplicável;
- revisão da paginação para aderência ao contrato padrão.

### 5.2 Fora do escopo
Esta User Story não contempla:

- criação de novos endpoints;
- versionamento avançado da API;
- catálogo corporativo de erros;
- internacionalização de mensagens;
- política completa de autenticação/autorização enterprise;
- observabilidade distribuída completa.

---

## 6. Descrição funcional

A aplicação deverá responder de forma consistente em todos os endpoints relevantes.

Isso inclui:

- sucesso com contrato previsível;
- paginação com shape único;
- erro de validação com status correto;
- recurso inexistente com status correto;
- erro de negócio ou processamento com tratamento consistente;
- erro inesperado sem vazamento bruto.

O consumidor da API não pode precisar adivinhar se um endpoint retorna:

- `data` em um lugar e `result` em outro;
- `message` num endpoint e `errors` em outro sem critério;
- `404` num caso e `200` com `null` em outro;
- stack trace ou objeto interno acidentalmente.

A API deve falar uma língua só.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

A implementação deve consolidar o comportamento HTTP no ponto central adequado do projeto.

Nada de cada controller inventar tratamento próprio para erro previsível.

Se já existe middleware/filtro/handler global, ele deve ser finalizado e adotado de verdade. Se existir parcialmente e estiver frouxo, deve ser corrigido.

### 7.2 Contrato mínimo de sucesso

As respostas de sucesso devem seguir padrão único e coerente com o projeto.

Isso inclui, quando aplicável:

- payload principal;
- metadados de paginação;
- identificadores auxiliares relevantes;
- ausência de campos desnecessários ou inconsistentes.

Não é obrigatório inventar envelope excessivo se o projeto for mais direto, mas a consistência é obrigatória.

### 7.3 Contrato mínimo de erro

As respostas de erro devem seguir um formato único, contendo no mínimo informações úteis e seguras, como:

- tipo/código do erro, quando existir no projeto;
- mensagem compreensível;
- detalhes de validação, quando aplicável;
- `requestId` ou correlação equivalente;
- timestamp, se fizer parte do padrão atual.

Sem exagero. Mas sem bagunça.

### 7.4 Mapeamento de status HTTP

O comportamento mínimo esperado é:

- **400** para entrada inválida ou malformed request;
- **404** para recurso não encontrado;
- **422** para falhas de validação/negócio quando essa distinção fizer parte do projeto;
- **500** para erro inesperado real.

O ponto aqui não é decorar HTTP. O ponto é parar de usar 500 para tudo.

### 7.5 Validação

Erros de validação devem ser tratados de forma estruturada e previsível.

A resposta deve permitir que quem consome a API entenda:

- qual campo ou parâmetro falhou;
- qual regra foi quebrada;
- se o erro é corrigível pelo cliente.

### 7.6 Não encontrado

Consultar uma transação inexistente, uma parcela inexistente ou um pagador inexistente não deve virar 500 nem 200 com payload vazio genérico.

Tem que virar o status correto com contrato consistente.

### 7.7 Erro inesperado

Erro interno inesperado deve:

- ser logado internamente;
- gerar resposta segura;
- carregar correlação útil;
- nunca expor stack trace bruto, SQL cru, segredo ou detalhe interno sensível.

### 7.8 Correlação

As respostas de erro, e preferencialmente o fluxo HTTP em geral, devem manter vínculo com `requestId` ou equivalente já adotado no projeto.

Sem correlação, troubleshooting vira loteria.

### 7.9 Compatibilidade com Swagger e testes

O contrato fechado aqui deve ser o mesmo usado para:

- documentação OpenAPI;
- testes E2E;
- validação final da entrega.

Se cada um falar uma verdade diferente, o projeto fica incoerente.

---

## 8. Critérios de aceite

### 8.1 Critérios funcionais
- Todos os endpoints obrigatórios respondem com contrato consistente.
- A paginação segue um formato único.
- Erros de entrada inválida retornam status coerente.
- Recursos inexistentes retornam 404 consistente.
- Falhas de validação/negócio retornam contrato previsível.
- Erros inesperados retornam resposta segura e padronizada.

### 8.2 Critérios técnicos
- Existe tratamento global de erro de verdade.
- O mapeamento de erro para status HTTP está centralizado e coerente.
- Não há controllers inventando respostas arbitrárias para erro.
- O contrato é reaproveitável em Swagger e testes.
- Não há quebra das US anteriores já concluídas.

### 8.3 Critérios de segurança
- Respostas de erro não expõem stack trace bruto.
- Respostas de erro não expõem SQL, segredo ou detalhes internos sensíveis.
- Dados sensíveis continuam protegidos nos payloads de sucesso e erro.

---

## 9. Evidências esperadas

- chamadas reais retornando sucesso com contrato consistente;
- chamada com parâmetro inválido retornando erro estruturado;
- chamada para recurso inexistente retornando 404 consistente;
- evidência de erro inesperado retornando 500 seguro;
- testes automatizados cobrindo os principais cenários de contrato e erro.

---

## 10. Riscos e observações

- Se cada endpoint continuar tratando erro do seu jeito, a API fica amadora.
- Se 500 continuar sendo usado como resposta padrão para tudo, o avaliador percebe rápido.
- Se stack trace vazar, a segurança da solução perde credibilidade.
- Se o contrato de paginação variar, Swagger e E2E vão sofrer depois.
- Se o `requestId` não estiver consistente, troubleshooting vai ficar pior do que precisa.

---

## 11. Definição de pronto

A User Story será considerada pronta quando:

- os endpoints obrigatórios seguirem um contrato HTTP consistente;
- o tratamento global de erro estiver fechado e centralizado;
- os principais status HTTP estiverem mapeados corretamente;
- as respostas forem seguras e previsíveis;
- os testes relevantes passarem;
- a base estiver pronta para Swagger e E2E sem retrabalho de contrato.

---

## 12. Observação final

Esta User Story não é perfumaria de API. Ela fecha a disciplina mínima que separa endpoint improvisado de interface confiável.

Se isso ficar frouxo, todo o resto perde força.
