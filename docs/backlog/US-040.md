# US-040 — Implementar infraestrutura real de persistência no SQL Server

## 1. Identificação

- **ID:** US-040
- **Título:** Implementar infraestrutura real de persistência no SQL Server
- **Épico:** Persistência, sincronização e API
- **Tipo:** Backend / Infraestrutura / Persistência
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-001 — Estruturar o projeto base da aplicação; US-002 — Definir a arquitetura modular e convenções de desenvolvimento; US-003 — Implantar convenção de enums, constantes, event types e error codes; US-004 — Configurar gerenciamento de configuração e segredos; US-006 — Modelar o banco de dados transacional no SQL Server; US-007 — Modelar tabelas de auditoria, rastreabilidade e evidência de integração; US-008 — Modelar tabelas de conflitos, rejeições e reconciliação; US-032 — Implementar persistência transacional com Outbox; US-033 — Implementar dispatcher de Outbox com retry e dead-letter; US-034 — Implementar Inbox para consumo idempotente de eventos internos; US-038 — Implementar mock server dos PSPs para desenvolvimento e teste; US-039 — Implementar testes unitários do domínio e dos adapters

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** implementar a infraestrutura real de persistência no SQL Server  
**para** permitir que a aplicação deixe de operar apenas com portas e stubs e passe a ter base concreta para sincronização real, consulta real, rastreabilidade real e testes integrados confiáveis.

---

## 3. Contexto de negócio

Até aqui o projeto já possui boa base de domínio, adapters, mock server, outbox conceitual e cobertura unitária relevante. Isso ajuda, mas não resolve o problema central do desafio: sem persistência real, a solução ainda não fecha o ciclo exigido pelo edital.

Hoje o maior risco não é falta de teste unitário. O maior risco é parecer pronto sem estar pronto. Enquanto a aplicação não possuir infraestrutura concreta de persistência no SQL Server, continuam abertos problemas como:

- sincronização sem efeito persistido real;
- checkpoints sem evolução confiável em banco;
- API sem fonte de consulta real;
- impossibilidade de validar idempotência de ponta a ponta;
- outbox e inbox sem backing store concreto;
- ambiente local subindo, mas sem comportamento sistêmico verdadeiro.

Esta User Story existe para fechar essa lacuna. O objetivo aqui não é ainda concluir toda a sincronização funcional nem todos os endpoints. O objetivo é montar a fundação concreta de persistência para que o restante deixe de ser especulação e passe a ser implementação real.

---

## 4. Objetivo

Implementar a infraestrutura real de persistência no SQL Server, contemplando:

- conexão real da aplicação com SQL Server;
- configuração segura e validada de acesso ao banco;
- bootstrap do schema do projeto em ambiente local/teste;
- implementação concreta de infraestrutura para acesso transacional;
- implementação concreta dos repositórios exigidos pelas portas atuais;
- base reutilizável para sincronização, leitura da API, outbox, inbox e testes de integração.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- inclusão e configuração do driver de SQL Server no projeto;
- definição da infraestrutura de conexão e pool;
- definição de estratégia de bootstrap do schema usando os scripts já existentes do projeto;
- implementação de `UnitOfWork` real para transações de banco;
- implementação concreta dos repositórios necessários para os fluxos já modelados nas portas da aplicação;
- integração da infraestrutura de persistência ao container/bootstrap da aplicação;
- organização da configuração de banco para ambiente local e de teste;
- documentação técnica mínima da estratégia adotada.

### 5.2 Fora do escopo
Esta User Story não contempla:

- conclusão da persistência do fluxo completo de sincronização;
- implementação dos endpoints REST obrigatórios;
- testes end-to-end completos;
- refatoração ampla do domínio sem necessidade;
- troca de SQL Server por outro banco;
- otimizações avançadas de performance ou tuning fino de query.

---

## 6. Descrição funcional

A aplicação deverá passar a possuir infraestrutura concreta para operar com SQL Server de forma previsível em ambiente local e de teste.

Ao final desta User Story, o projeto deverá ser capaz de:

- abrir conexão real com o banco configurado;
- inicializar ou preparar o schema necessário ao projeto usando os artefatos SQL já presentes no repositório;
- disponibilizar implementações reais das portas de persistência já definidas;
- executar operações transacionais de forma controlada;
- permitir que as próximas User Stories implementem sincronização e consulta real sem depender de stubs temporários.

O foco aqui é preparar a fundação. Se essa fundação ficar errada, o resto do projeto vai desmoronar ou virar gambiarra.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

A implementação deve ser pragmática, simples e compatível com a arquitetura já existente. Não é para inventar ORM pesado nem abstração cosmética sem valor.

A infraestrutura deverá:

- respeitar a separação entre aplicação e infraestrutura;
- atender as portas já existentes, sem reescrever o domínio à toa;
- usar SQL Server como banco oficial do projeto;
- ser adequada para execução local, testes de integração e uso via Docker Compose;
- permitir transação explícita onde o fluxo exigir atomicidade.

### 7.2 Componentes mínimos esperados

A solução deverá incluir, no mínimo:

#### a) Configuração de banco
- leitura de variáveis de ambiente obrigatórias;
- validação de configuração inválida na inicialização;
- montagem segura da connection string;
- separação clara entre ambiente local/mock/teste.

#### b) Infraestrutura de conexão
- componente responsável por criar e fornecer conexões SQL Server;
- estratégia clara para reutilização/pool;
- falha explícita quando o banco não estiver acessível.

#### c) Bootstrap de schema
- mecanismo para preparar o banco com os scripts SQL versionados já existentes no projeto;
- ordem garantida de execução dos scripts;
- possibilidade de uso em ambiente local e em testes.

#### d) Unidade transacional
- implementação concreta de `UnitOfWork`;
- suporte a begin, commit e rollback;
- integração com operações que precisem persistir estado e outbox de forma consistente.

#### e) Repositórios concretos
A infraestrutura deverá prover implementações reais, no mínimo, para as portas já existentes que sustentam o núcleo da aplicação, como:

- repositórios de consulta de transações;
- repositórios de consulta de parcelas;
- repositório de checkpoint de sincronização;
- repositório de idempotência, se a porta atual exigir backing store persistente;
- repositório de outbox;
- repositório de inbox.

### 7.3 Integração com o container da aplicação

O bootstrap da aplicação deverá deixar de operar apenas com config solta e passar a registrar as dependências concretas de persistência.

Isso inclui:

- wiring dos componentes de infraestrutura no container;
- disponibilização das implementações para os use cases e serviços existentes;
- eliminação de lacunas onde hoje só existe porta sem implementação real.

### 7.4 Estratégia de schema

O projeto já possui scripts SQL versionados para criação de database, tabelas core, tabelas operacionais, auditoria, conflitos, rejeições, reconciliação e índices.

Esta User Story deverá definir como esses scripts serão usados pela aplicação e pelos testes, sem criar um segundo caminho concorrente desnecessário.

A regra aqui é simples:

- aproveitar o que já existe;
- padronizar a execução;
- evitar duplicação de mecanismo.

### 7.5 Restrições técnicas

A implementação não deverá:

- introduzir dependência de banco diferente de SQL Server;
- inventar repositórios ou tabelas fora do que o projeto já modelou sem justificativa real;
- acoplar domínio a detalhes de driver;
- esconder erro de conexão com fallback silencioso;
- criar camada genérica exagerada que só aumente complexidade.

---

## 8. Critérios de aceitação

### 8.1 Critérios funcionais
- A aplicação deve conseguir abrir conexão real com SQL Server usando configuração externa válida.
- O projeto deve possuir mecanismo reproduzível para preparar o schema do banco usando os scripts versionados existentes.
- O container/bootstrap da aplicação deve registrar implementações reais das portas de persistência necessárias.
- O projeto deve possuir `UnitOfWork` real com suporte a transação explícita.
- O projeto deve disponibilizar repositórios concretos para consulta e componentes operacionais já previstos na arquitetura.

### 8.2 Critérios técnicos
- O `package.json` deve conter as dependências necessárias para SQL Server.
- A falha de configuração de banco deve ser detectada de forma explícita e previsível.
- A infraestrutura de persistência não deve quebrar os testes unitários já existentes.
- O código de infraestrutura deve ficar isolado em camada coerente com a arquitetura modular.
- Não deve haver credencial hardcoded no repositório.

### 8.3 Critérios operacionais
- O ambiente local deve conseguir subir com banco disponível para uso do projeto.
- Deve existir forma clara de preparar o banco em ambiente local/teste.
- A documentação mínima da configuração do banco deve ser atualizada.

---

## 9. Evidências esperadas

Ao concluir esta User Story, devem existir evidências objetivas como:

- dependência de SQL Server adicionada ao projeto;
- wiring real de persistência no bootstrap;
- componente de conexão implementado;
- `UnitOfWork` implementado;
- repositórios concretos implementados;
- mecanismo documentado de preparação do schema;
- execução local comprovando que a aplicação conecta no banco corretamente.

---

## 10. Riscos e pontos de atenção

- Implementar apenas conexão sem fechar wiring útil não resolve o problema.
- Criar infraestrutura genérica demais vai atrasar sem agregar valor.
- Duplicar mecanismo de schema vai gerar caos operacional.
- Misturar nesta US regras de negócio da sincronização vai ampliar escopo e atrasar a entrega.
- Quebrar os testes unitários já existentes por acoplamento indevido será regressão inaceitável.

---

## 11. Definição de pronto

Esta User Story será considerada pronta quando:

- a aplicação possuir infraestrutura concreta de persistência em SQL Server;
- o bootstrap do banco estiver definido e reproduzível;
- as portas de persistência prioritárias já tiverem implementações reais disponíveis no container;
- a base estiver preparada para a próxima User Story de persistência real da sincronização;
- o projeto continuar compatível com o que já foi implementado nas US anteriores.
