# US-048 — Implementar health/readiness e observabilidade operacional mínima

## 1. Identificação

- **ID:** US-048
- **Título:** Implementar health/readiness e observabilidade operacional mínima
- **Épico:** Operação local, confiabilidade e suporte
- **Tipo:** Backend / Observabilidade / Operação
- **Prioridade:** Média
- **Status inicial:** Backlog
- **Dependências:** US-010 — Implantar logging estruturado com requestId e correlação; US-012 — Implementar resiliência de integração externa; US-031 — Implementar observabilidade e métricas de cache; US-042 — Implementar mecanismo acionável de sincronização de verdade; US-043 — Implementar os endpoints REST obrigatórios com consulta real; US-044 — Fechar o contrato HTTP e tratamento global de erro; US-045 — Integrar Docker Compose com bootstrap funcional da aplicação

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** implementar health/readiness e observabilidade operacional mínima  
**para** expor o estado operacional da aplicação de forma objetiva, facilitar troubleshooting e deixar o ambiente local mais confiável para validação, testes e demonstração.

---

## 3. Contexto de negócio

Depois que a aplicação passa a ter banco real, sync real, API real e ambiente Compose funcional, surge um problema óbvio: como saber se ela está viva, pronta e minimamente observável sem ficar adivinhando por log solto?

Esse é o tipo de falha operacional idiota que derruba demonstração e confunde troubleshooting:

- processo subiu, mas banco não conecta;
- app responde em uma rota, mas não está pronta para operar;
- sync falha e ninguém sabe onde olhar;
- latência e erro ficam invisíveis;
- Compose marca container como “up”, mas o serviço ainda está quebrado.

O objetivo aqui não é montar plataforma enterprise de observabilidade. É fechar o mínimo que faz diferença real:

- liveness;
- readiness;
- sinais operacionais básicos;
- logs úteis;
- visibilidade mínima do comportamento da aplicação.

---

## 4. Objetivo

Implementar health/readiness e observabilidade operacional mínima, contemplando:

- endpoint de liveness;
- endpoint de readiness com dependências críticas;
- logs estruturados úteis para operação local;
- métricas ou sinais mínimos úteis para troubleshooting;
- integração pragmática com o ambiente Docker/local.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- implementação de `GET /health`;
- implementação de `GET /ready`;
- verificação de prontidão da aplicação e dependências críticas;
- consolidação mínima de logs estruturados operacionais;
- exposição mínima de métricas ou indicadores úteis, se isso já estiver viável no projeto;
- uso desses sinais no ambiente local/Compose.

### 5.2 Fora do escopo
Esta User Story não contempla:

- APM enterprise completo;
- tracing distribuído full;
- dashboard completo;
- alerting corporativo;
- service mesh;
- telemetria exagerada sem uso prático.

---

## 6. Descrição funcional

A aplicação deverá disponibilizar, no mínimo:

- `GET /health`
- `GET /ready`

### 6.1 Health
O endpoint de health deve responder rapidamente se o processo está vivo e executando.

Não é para consultar tudo. É para responder se a aplicação está de pé.

### 6.2 Ready
O endpoint de readiness deve responder se a aplicação está pronta para operar no contexto real do projeto.

Isso inclui, no mínimo, verificar as dependências críticas necessárias para o fluxo principal, como por exemplo:

- conectividade com banco;
- capacidade mínima de uso da configuração ativa;
- dependências realmente necessárias ao funcionamento do serviço.

### 6.3 Observabilidade mínima
A aplicação deve emitir sinais úteis para operação local, como por exemplo:

- logs estruturados com correlação;
- identificação clara de falhas de inicialização;
- registro útil de execução de sync;
- registro útil de erros HTTP relevantes;
- métricas mínimas já viáveis no projeto, sem inflar escopo.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

A implementação deve ser pragmática. O objetivo não é parecer sofisticado. O objetivo é parar de operar no escuro.

### 7.2 Liveness vs readiness

A distinção é obrigatória:

- **liveness**: processo vivo;
- **readiness**: serviço pronto.

Se os dois fizerem exatamente a mesma coisa, a implementação está conceitualmente errada.

### 7.3 Endpoint `/health`

O endpoint deve:

- responder rápido;
- não depender de checagens profundas;
- ser seguro e simples;
- servir como verificação de vida do processo.

### 7.4 Endpoint `/ready`

O endpoint deve:

- validar dependências críticas reais;
- falhar quando o serviço não puder operar corretamente;
- expor resposta objetiva e segura;
- ser utilizável por Compose, scripts e validação manual.

### 7.5 Dependências críticas

A readiness deve verificar o que realmente importa para este projeto. No mínimo:

- banco SQL Server acessível;
- configuração essencial resolvida;
- estado mínimo que permita API e sync operarem.

Não vale inventar dependência irrelevante só para deixar bonito.

### 7.6 Logs operacionais

Os logs mínimos precisam ser úteis de verdade. Isso inclui, quando aplicável:

- requestId/correlationId;
- erro de integração externa;
- falha de readiness;
- início/fim de sincronização;
- contagens resumidas de processamento;
- erro global HTTP sem vazamento indevido.

### 7.7 Métricas e sinais mínimos

Se a infraestrutura atual já permitir, expor o mínimo útil, como por exemplo:

- contagem de requests;
- latência básica;
- erros HTTP;
- execuções de sync;
- falhas de sync.

Se isso ainda não estiver maduro no código, priorizar primeiro o que é obrigatório: health, ready e logs úteis. Não invente métrica fake.

### 7.8 Compose e ambiente local

A implementação deve ser utilizável pelo ambiente local, por exemplo para:

- healthcheck de container;
- verificação manual de prontidão;
- validação rápida antes de rodar sync;
- apoio à demonstração.

---

## 8. Critérios de aceite

### 8.1 Critérios funcionais
- A aplicação expõe `GET /health`.
- A aplicação expõe `GET /ready`.
- `GET /health` responde liveness do processo.
- `GET /ready` responde prontidão real do serviço.
- Falha de dependência crítica impacta readiness corretamente.
- Logs mínimos úteis existem para operação e troubleshooting.

### 8.2 Critérios técnicos
- Liveness e readiness têm comportamentos distintos.
- Readiness verifica dependências críticas reais.
- A resposta é simples, segura e consistente.
- A implementação é compatível com Docker Compose/local.
- Não há quebra das US anteriores já concluídas.

### 8.3 Critérios operacionais
- O time consegue validar rapidamente se o ambiente está pronto.
- Falhas de inicialização e dependência ficam identificáveis.
- A solução deixa de depender exclusivamente de leitura manual confusa de log.

---

## 9. Evidências esperadas

- chamada real a `/health`;
- chamada real a `/ready`;
- evidência de `/ready` falhando quando dependência crítica estiver indisponível;
- evidência de logs úteis durante startup, erro e sync;
- evidência de uso dos endpoints no ambiente local/Compose.

---

## 10. Riscos e observações

- Se `/ready` fizer a mesma coisa que `/health`, foi feito errado.
- Se readiness ignorar o banco, o sinal operacional fica mentiroso.
- Se os logs continuarem vagos, troubleshooting continua ruim.
- Se tentarem inflar telemetria antes do básico funcionar, vão desperdiçar tempo.
- Se a resposta expuser detalhe interno sensível, você troca um problema por outro.

---

## 11. Definição de pronto

A User Story será considerada pronta quando:

- health e readiness estiverem implementados corretamente;
- a aplicação fornecer sinais mínimos úteis de operação;
- os endpoints funcionarem no ambiente local/Compose;
- falhas críticas refletirem em readiness de forma coerente;
- a base estiver mais segura para testes, demo e fechamento final.

---

## 12. Observação final

Esta User Story fecha o mínimo operacional necessário para parar de testar e demonstrar a solução no escuro.

Sem isso, qualquer falha simples vira ruído, perda de tempo e risco desnecessário.
