# US-028 — Implementar contrato padronizado de resposta e paginação

## 1. Identificação

- **ID:** US-028
- **Título:** Implementar contrato padronizado de resposta e paginação
- **Épico:** API de consulta
- **Tipo:** API / Arquitetura / Padronização
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-001 — Estruturar o projeto base da aplicação; US-002 — Definir a arquitetura modular e convenções de desenvolvimento; US-003 — Implantar convenção de enums, constantes, event types e error codes; US-004 — Configurar gerenciamento de configuração e segredos; US-010 — Implantar logging estruturado com requestId e correlação; US-011 — Padronizar tratamento de erros da aplicação; US-023 — Implementar endpoint de listagem paginada de transações; US-024 — Implementar endpoint de detalhe de transação; US-025 — Implementar endpoint de listagem de parcelas da transação; US-026 — Implementar endpoint de detalhe de parcela; US-027 — Implementar endpoint de consulta de pagador da transação

---

## 2. História

**Como** time de engenharia responsável pela API de consulta do agregador multi-PSP  
**quero** implementar um contrato padronizado de resposta e paginação  
**para** garantir consistência entre endpoints, previsibilidade para consumidores, redução de ambiguidade no uso da API e base estável para documentação, testes, cache e evolução futura.

---

## 3. Contexto de negócio

A solução deverá disponibilizar endpoints de consulta capazes de atender operação, auditoria e validação funcional, com comportamento consistente e seguro. O desafio define explicitamente metadados de paginação para o endpoint `GET /transactions`, incluindo `total`, `page`, `limit` e `totalPages`, além de exigir respostas padronizadas para erros como `400`, `404`, `422` e `500`. fileciteturn13file0

Sem padronização, cada endpoint tende a retornar formatos ligeiramente distintos, o que aumenta:
- custo de consumo da API;
- inconsistência de testes;
- risco de documentação divergente;
- esforço de manutenção e cache;
- dificuldade de observabilidade e troubleshooting.

A padronização do contrato não é perfumaria. É requisito de maturidade da API.

---

## 4. Objetivo

Implementar o padrão de resposta da API, contemplando:

- envelope consistente para respostas paginadas;
- envelope consistente para respostas de recurso único;
- contrato padronizado para erros;
- metadados de paginação reutilizáveis;
- semântica consistente de ausência de resultado;
- integração com requestId, tratamento global de erros e documentação.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- definição do padrão de resposta de sucesso para coleções paginadas;
- definição do padrão de resposta de sucesso para recursos únicos;
- definição do padrão de resposta de erro;
- definição dos metadados obrigatórios de paginação;
- criação de DTOs/utilitários reutilizáveis de paginação;
- integração do padrão com os endpoints já definidos;
- documentação técnica do contrato padronizado.

### 5.2 Fora do escopo
Esta User Story não contempla:

- versionamento público da API;
- paginação cursor-based para a v1;
- padronização de contratos de webhook;
- internacionalização de mensagens de erro;
- HAL, JSON:API ou outro protocolo formal externo;
- reescrita do modelo de domínio para se adequar ao contrato HTTP.

---

## 6. Descrição funcional

A API deverá responder de forma consistente em todos os endpoints de consulta. O consumidor deverá conseguir prever:

- onde está o dado principal da resposta;
- onde estão os metadados de paginação;
- como identificar erros e correlacioná-los com o requestId;
- como interpretar coleções vazias;
- como tratar recursos únicos versus listas.

O padrão deverá reduzir variação entre endpoints e tornar a experiência da API previsível.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

A implementação deverá definir contratos reutilizáveis, desacoplados do domínio, específicos da camada de apresentação HTTP. O objetivo é evitar que cada controller monte manualmente sua própria estrutura de resposta.

O padrão deverá contemplar:

- resposta paginada;
- resposta de recurso único;
- resposta de erro.

### 7.2 Tipos de resposta padronizada

A API deverá suportar, no mínimo, os seguintes formatos.

#### a) Resposta paginada
Utilizada em endpoints de coleção, como `GET /transactions`.

Estrutura mínima esperada:
- `data`: coleção de itens;
- `pagination`: metadados de paginação.

#### b) Resposta de recurso único
Utilizada em endpoints de detalhe, como:
- `GET /transactions/:id`
- `GET /transactions/:transactionId/installments/:installmentId`
- `GET /transactions/:transactionId/payers`

Estrutura mínima esperada:
- `data`: objeto do recurso solicitado.

#### c) Resposta de erro
Utilizada para falhas padronizadas.

Estrutura mínima esperada:
- `timestamp`
- `status`
- `code`
- `message`
- `requestId`

Essa estrutura deverá ser coerente com a política global de tratamento de erros da aplicação.

### 7.3 Contrato de paginação

O contrato de paginação deverá conter, no mínimo:

- `total`
- `page`
- `limit`
- `totalPages`

Esses campos estão explicitamente previstos no desafio para a listagem de transações. fileciteturn13file0

### 7.4 Regras de paginação

A implementação deverá padronizar, no mínimo:

- `page` como índice iniciado em 1;
- `limit` como tamanho da página;
- `totalPages` calculado a partir de `total` e `limit`;
- coerência para lista vazia com paginação válida;
- resposta consistente mesmo quando `total = 0`.

### 7.5 Semântica de coleções vazias

Quando uma coleção não possuir resultados:
- a resposta deverá retornar `data: []`;
- a paginação deverá permanecer válida;
- a ausência de resultado não deverá ser tratada como erro.

### 7.6 Semântica de recurso único

Quando um recurso único for encontrado:
- a resposta deverá retornar `data` com o objeto correspondente.

Quando não for encontrado:
- a API deverá responder com erro padronizado `404`.

A ausência de recurso único não deverá resultar em `data: null` com `200`.

### 7.7 Semântica do erro

O contrato de erro deverá ser estável e alinhado ao tratamento global da aplicação. No mínimo:

- `400` para parâmetros inválidos;
- `404` para recurso não encontrado;
- `422` para falhas de validação de negócio;
- `500` para erro interno sem vazamento de detalhes. fileciteturn13file0

### 7.8 Integração com requestId

Toda resposta de erro deverá conter `requestId` para rastreabilidade operacional. Em respostas de sucesso, a presença desse identificador poderá ser feita por header ou por observabilidade correlata, conforme a convenção do projeto. A política escolhida deverá ser documentada.

### 7.9 Integração com controllers e mappers

A implementação deverá evitar montagem manual e repetitiva de envelopes de resposta em cada controller. A base deverá prever:

- utilitários de resposta paginada;
- utilitários de resposta de recurso único;
- mappers HTTP consistentes;
- schemas de resposta compatíveis com o contrato padrão.

### 7.10 Integração com documentação OpenAPI

A padronização desta User Story deverá facilitar a documentação posterior da API, permitindo reutilização de schemas para:
- `PaginatedResponse<T>`
- `ErrorResponse`
- `ResourceResponse<T>`

A implementação não precisa publicar Swagger nesta história, mas deverá preparar a base para isso.

### 7.11 Integração com cache futuro

O contrato padronizado deverá ser compatível com cache em camada HTTP/aplicação, sem ambiguidade estrutural. A previsibilidade da resposta é essencial para:
- cache de listagens;
- cache de recursos únicos;
- invalidação e reidratação futura.

### 7.12 Estruturas mínimas esperadas

A base deverá prever, no mínimo, os seguintes componentes:

#### a) `PaginatedResponse<T>`
Estrutura genérica para coleções paginadas.

#### b) `PaginationMeta`
Estrutura de metadados de paginação.

#### c) `ResourceResponse<T>`
Estrutura genérica para recurso único.

#### d) `ErrorResponse`
Estrutura padronizada de erro HTTP.

#### e) `PaginationQuery`
Estrutura reutilizável de entrada para paginação, quando aplicável.

## 8. Regras e princípios

A implementação desta User Story deverá obedecer aos seguintes princípios:

- consistência entre endpoints;
- previsibilidade do contrato;
- separação entre domínio e envelope HTTP;
- erro padronizado e rastreável;
- paginação uniforme;
- compatibilidade com cache e OpenAPI futuros;
- ausência de variação estrutural arbitrária entre controllers.

## 10. Critérios de aceite

### 10.1 Critérios funcionais
- Endpoints de consulta deverão responder com contrato consistente.
- Endpoints paginados deverão retornar `data` e `pagination`.
- Respostas de erro deverão seguir contrato padronizado.
- Coleções vazias deverão retornar lista vazia com paginação coerente.
- Recursos não encontrados deverão retornar `404` padronizado.

### 10.2 Critérios técnicos
- O contrato deverá ser reutilizável por múltiplos endpoints.
- `PaginationMeta` deverá conter `total`, `page`, `limit` e `totalPages`.
- `ErrorResponse` deverá conter `timestamp`, `status`, `code`, `message` e `requestId`.
- A implementação deverá estar desacoplada do domínio e concentrada na camada HTTP/apresentação.
- A estrutura deverá ser compatível com documentação OpenAPI futura.
- Os controllers não deverão montar envelopes arbitrários por conta própria.

### 10.3 Critérios de qualidade
- A implementação deverá ser legível, modular e testável.
- A documentação deverá explicar claramente a semântica de cada tipo de resposta.
- A padronização deverá reduzir ambiguidade para consumidores e para o time.
- A base criada deverá facilitar cache, testes e evolução da API.

## 11. Mapeamento de testes

### 11.1 Testes unitários
- Validar cálculo de `totalPages`.
- Validar serialização de `PaginatedResponse`.
- Validar serialização de `ResourceResponse`.
- Validar estrutura mínima de `ErrorResponse`.

### 11.2 Testes de integração
- Validar aplicação do contrato padronizado no endpoint de listagem.
- Validar aplicação do contrato padronizado em endpoint de recurso único.
- Validar resposta de erro com `requestId`.
- Validar comportamento consistente para coleção vazia.

### 11.3 Testes de qualidade técnica
- Validar coerência entre documentação e contratos implementados.
- Validar ausência de envelopes divergentes entre controllers.
- Validar aderência do padrão à política global de erros da aplicação.

### 11.4 Evidências esperadas
- DTOs padronizados de resposta presentes no projeto;
- endpoints usando o contrato padronizado;
- documentação técnica do contrato presente;
- testes cobrindo paginação, recurso único e erro padronizado.
