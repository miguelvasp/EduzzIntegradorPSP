# US-049 — Consolidar README e documentação técnica enxuta

## 1. Identificação

- **ID:** US-049
- **Título:** Consolidar README e documentação técnica enxuta
- **Épico:** Documentação final e sustentação da entrega
- **Tipo:** Documentação / Operação / Governança técnica
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-042 — Implementar mecanismo acionável de sincronização de verdade; US-043 — Implementar os endpoints REST obrigatórios com consulta real; US-044 — Fechar o contrato HTTP e tratamento global de erro; US-045 — Integrar Docker Compose com bootstrap funcional da aplicação; US-047 — Publicar Swagger/OpenAPI em `/docs`; US-048 — Implementar health/readiness e observabilidade operacional mínima

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** consolidar um README operacional forte e uma documentação técnica enxuta  
**para** garantir que a solução possa ser executada, entendida e avaliada sem atrito, sem virar uma bíblia inútil e sem depender de explicação oral complementar.

---

## 3. Contexto de negócio

Documentação ruim derruba solução boa. Documentação inflada também.

O desafio exige README com instruções objetivas de execução, testes, sincronização, variáveis de ambiente e decisões de arquitetura. Ao mesmo tempo, encher o repositório de documento gigante e redundante é desperdício e piora a leitura. O caminho certo aqui é documentação **curta, executável e honesta**.

Depois das US anteriores, a aplicação já deve ter:

- ambiente Compose funcional;
- sync acionável;
- endpoints obrigatórios;
- contrato HTTP fechado;
- Swagger publicado;
- health/readiness.

Agora faz sentido consolidar a documentação final do projeto, mas de forma enxuta, conectada ao que realmente existe e útil para quem vai avaliar.

---

## 4. Objetivo

Consolidar o README e a documentação técnica enxuta, contemplando:

- instruções claras para subir o ambiente;
- instruções claras para rodar sincronização;
- instruções claras para testar a API;
- instruções claras para executar testes automatizados;
- variáveis de ambiente relevantes;
- visão técnica curta da arquitetura e dos principais fluxos.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- consolidação do `README.md` principal;
- organização de documentação técnica curta em pasta versionada;
- documentação do fluxo principal de uso da aplicação;
- documentação dos comandos essenciais;
- documentação das variáveis de ambiente relevantes;
- documentação curta das decisões arquiteturais que realmente importam;
- documentação alinhada ao estado real da solução.

### 5.2 Fora do escopo
Esta User Story não contempla:

- documentação corporativa extensa;
- apostila técnica;
- tutorial inchado;
- duplicação do Swagger dentro do README;
- texto acadêmico sobre arquitetura;
- documento mentiroso sobre features que não existem.

---

## 6. Descrição funcional

A documentação deve permitir que um avaliador ou desenvolvedor consiga, sem conversa adicional:

1. entender o que a solução faz;
2. subir o ambiente local;
3. verificar que a app está pronta;
4. disparar a sincronização;
5. consultar a API;
6. abrir o Swagger;
7. rodar os testes;
8. localizar rapidamente as decisões técnicas principais.

O README deve ser direto. A documentação técnica complementar deve existir só onde ela realmente agrega.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

A documentação deve seguir três princípios:

- **curta**
- **executável**
- **verdadeira**

Se um trecho não ajuda a rodar, validar, entender ou defender a solução, ele provavelmente é excesso.

### 7.2 Conteúdo mínimo do README

O `README.md` deve conter, no mínimo:

- visão geral do desafio e da solução;
- stack principal;
- como subir com Docker Compose;
- como verificar health/readiness;
- como acionar a sincronização;
- como consultar os endpoints principais;
- onde acessar `/docs`;
- como rodar testes;
- principais variáveis de ambiente;
- resumo das decisões arquiteturais.

### 7.3 Estrutura recomendada

Estrutura enxuta recomendada:

- `README.md`
- `docs/architecture.md` ou equivalente curto
- `docs/testing.md` se realmente necessário
- `docs/demo.md` ou equivalente curto, se fizer sentido mais à frente

Não criar cinco documentos para dizer a mesma coisa.

### 7.4 Arquitetura a documentar

Documentar só o que sustenta a solução real, por exemplo:

- módulos principais;
- sync acionável;
- persistência real no SQL Server;
- separação entre adapters PSP, domínio e API;
- idempotência;
- proteção do `document`;
- query layer e API de consulta;
- outbox/inbox somente se realmente estiverem ativos no fluxo final.

### 7.5 Variáveis de ambiente

As variáveis de ambiente relevantes devem ser listadas com clareza:

- nome;
- propósito;
- obrigatória ou opcional;
- exemplo seguro quando útil.

Nada de esconder variável crítica em arquivo obscuro e fingir que o README resolve.

### 7.6 Comandos e exemplos

Os comandos principais devem ser reproduzíveis e curtos, por exemplo:

- subir ambiente;
- derrubar ambiente;
- rodar sync;
- executar testes;
- acessar Swagger;
- chamar endpoints principais.

### 7.7 Limites e honestidade

Se algo for escolha consciente de escopo, isso pode ser documentado brevemente.

O que não pode:

- prometer feature não implementada;
- fingir que existe risco/fraude sofisticada se não existe;
- documentar arquitetura enterprise que o código não sustenta.

---

## 8. Critérios de aceite

### 8.1 Critérios funcionais
- Existe um `README.md` claro e executável.
- O README explica como subir, validar, sincronizar e testar a solução.
- Existe documentação técnica complementar curta e útil.
- Um avaliador consegue usar a solução sem depender de explicação oral extra.
- O conteúdo está alinhado ao comportamento real do projeto.

### 8.2 Critérios técnicos
- A documentação não está inchada.
- Não há duplicação inútil entre README, Swagger e docs.
- As variáveis de ambiente relevantes estão claramente descritas.
- As decisões arquiteturais principais estão registradas de forma honesta.
- Não há quebra das US anteriores já concluídas.

### 8.3 Critérios de qualidade
- A documentação é curta e útil.
- O texto não inventa feature.
- O texto não vira bíblia.
- O texto ajuda a avaliação técnica em vez de atrapalhar.

---

## 9. Evidências esperadas

- `README.md` consolidado;
- documentação técnica curta versionada no repositório;
- comandos principais descritos e testáveis;
- evidência de alinhamento entre documentação e execução real.

---

## 10. Riscos e observações

- Se o README ficar longo demais, ninguém lê.
- Se ficar curto demais e vazio, não serve.
- Se repetir Swagger no README, virou ruído.
- Se documentar feature inexistente, você perde credibilidade.
- Se esconder setup importante fora da documentação, a entrega fica frágil.

---

## 11. Definição de pronto

A User Story será considerada pronta quando:

- o README estiver consolidado e operacional;
- a documentação técnica curta estiver organizada;
- os comandos principais estiverem claros;
- as variáveis de ambiente estiverem bem descritas;
- o material estiver alinhado ao que a aplicação realmente faz;
- a solução ficar pronta para validação final e demonstração sem retrabalho documental.

---

## 12. Observação final

Esta User Story serve para fechar a documentação do jeito certo: forte o bastante para sustentar a entrega, enxuta o bastante para não virar peso morto.

Se você errar a mão aqui, ou fica amador ou vira prolixo.
