# US-046 — Implementar testes de integração críticos

## 1. Identificação

- **ID:** US-046
- **Título:** Implementar testes de integração críticos
- **Épico:** Qualidade e validação sistêmica
- **Tipo:** Qualidade / Integração / Persistência
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-040 — Implementar infraestrutura real de persistência no SQL Server; US-041 — Implementar persistência real da sincronização; US-042 — Implementar mecanismo acionável de sincronização de verdade; US-043 — Implementar os endpoints REST obrigatórios com consulta real; US-044 — Fechar o contrato HTTP e tratamento global de erro; US-045 — Integrar Docker Compose com bootstrap funcional da aplicação

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** implementar os testes de integração críticos da solução  
**para** validar o comportamento real entre banco, sincronização e API, reduzindo o risco de uma entrega bonita em unidade e quebrada na colaboração entre componentes.

---

## 3. Contexto de negócio

Agora sim faz sentido testar integração de verdade.

Antes disso, tentar fazer teste de integração era teatro, porque faltavam persistência real, sync real e API real. Depois das US-040 a US-045, o sistema finalmente passa a ter os blocos necessários para validar a colaboração real entre componentes.

O objetivo aqui não é encher o projeto de teste lento e decorativo. O objetivo é proteger os fluxos que realmente podem quebrar feio:

- sync persistindo errado;
- reimportação gerando duplicidade;
- checkpoint não evoluindo;
- API lendo estado inconsistente;
- contrato HTTP quebrando quando encontra dado real;
- integração entre app, banco e mock falhando fora do unit test.

Sem essa camada, o sistema continua com um buraco entre “regra local funciona” e “fluxo real funciona”.

---

## 4. Objetivo

Implementar a suíte de testes de integração críticos, contemplando:

- persistência real em banco controlado;
- execução real da sincronização;
- validação do efeito persistido;
- validação da leitura da API sobre o estado persistido;
- validação de idempotência e segurança nos fluxos principais.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- testes de integração entre sync e SQL Server;
- testes de integração entre sync persistida e API de consulta;
- testes de integração de reimportação/idempotência;
- testes de integração de checkpoint incremental;
- testes de integração de contrato HTTP em cenários reais;
- setup controlado de ambiente de teste para banco e mock quando necessário.

### 5.2 Fora do escopo
Esta User Story não contempla:

- suíte E2E completa de demonstração;
- testes de carga ou performance;
- explosão combinatória de cenários marginais;
- uso do mock server como dependência para testes unitários;
- cobertura redundante de regra já muito bem protegida em unit tests;
- caos engineering.

---

## 6. Descrição funcional

A suíte de integração deve provar que o sistema funciona quando as peças reais colaboram.

Os testes devem cobrir, no mínimo, estes grupos:

### 6.1 Sync + persistência
- executar sincronização real contra ambiente controlado;
- validar gravação de transações, parcelas e pagador;
- validar proteção do `document`;
- validar checkpoint persistido.

### 6.2 Reimportação / idempotência
- reexecutar o mesmo conjunto de dados;
- comprovar ausência de duplicidade;
- comprovar que merge/atualização segue o comportamento esperado.

### 6.3 API lendo dado real
- consultar os endpoints obrigatórios após persistência real;
- validar retorno coerente do banco;
- validar paginação e contratos em cenário real.

### 6.4 Erro controlado
- validar resposta consistente para recurso inexistente;
- validar comportamento diante de payload inválido ou parâmetro inválido;
- validar que erro real não vaza dado indevido.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

Esses testes devem existir para proteger colaboração real, não para repetir unit test em câmera lenta.

Se um teste não valida integração concreta entre componentes, ele provavelmente não deveria estar nesta camada.

### 7.2 Componentes sob integração

A suíte deve integrar de verdade, conforme o escopo do cenário:

- aplicação HTTP;
- camada de consulta;
- pipeline de sincronização;
- repositórios concretos;
- SQL Server de teste/controlado;
- mock server, somente quando necessário ao fluxo de integração real.

### 7.3 Cenários mínimos obrigatórios

#### 7.3.1 Sincronização persiste estado real
Validar que uma execução de sincronização grava corretamente:

- transação;
- parcelas;
- pagador;
- checkpoint.

#### 7.3.2 Reimportação não duplica
Validar que repetir a sincronização do mesmo item ou lote:

- não duplica transação;
- não duplica parcela;
- preserva chave de idempotência;
- mantém consistência do estado final.

#### 7.3.3 API consulta o que foi persistido
Após sincronização real, validar:

- `GET /transactions`
- `GET /transactions/:id`
- `GET /transactions/:id/installments`
- `GET /installments/:id`
- `GET /transactions/:id/payer`

Tudo sobre dado real persistido.

#### 7.3.4 Segurança do pagador
Validar que o `document`:

- não está em texto puro na base, conforme o desenho adotado;
- não aparece em texto puro na API.

#### 7.3.5 Erros previsíveis
Validar cenários como:

- ID inexistente retornando 404;
- parâmetro inválido retornando erro consistente;
- falha inesperada retornando 500 seguro, quando esse cenário for viável de testar sem teatro.

### 7.4 Estratégia de ambiente de teste

A camada de integração deve usar ambiente controlado e reproduzível.

Pode usar:

- banco real de teste;
- bootstrap automatizado;
- mock server controlado por cenário;
- setup/teardown pragmático.

O que não pode:

- depender de estado manual herdado;
- usar banco compartilhado sujo;
- transformar o teste em ritual frágil.

### 7.5 Critério de seleção dos testes

Priorizar somente o que protege risco real:

- persistência;
- reimportação;
- consulta real;
- segurança;
- contrato.

Não criar teste trivial só para inflar número.

### 7.6 Velocidade e manutenção

A suíte deve ser pequena, cirúrgica e confiável.

Se ficar lenta, frágil e redundante, você matou o valor dela.

---

## 8. Critérios de aceite

### 8.1 Critérios funcionais
- Existe suíte de integração crítica cobrindo sync, persistência e API.
- A sincronização real é validada contra banco controlado.
- A reimportação sem duplicidade é validada.
- Os endpoints obrigatórios são validados sobre dado persistido.
- A proteção do `document` é validada.
- Erros previsíveis relevantes são validados.

### 8.2 Critérios técnicos
- Os testes usam componentes reais das camadas integradas.
- Os testes não repetem desnecessariamente os unit tests.
- Os testes não usam mock server em testes unitários.
- O setup é reproduzível.
- Não há dependência de estado manual obscuro.
- Não há quebra das US anteriores já concluídas.

### 8.3 Critérios de qualidade
- A suíte é objetiva e tem valor real.
- Os testes falham quando a integração quebra de verdade.
- Os testes não são frágeis por detalhes irrelevantes.

---

## 9. Evidências esperadas

- execução da suíte de integração passando;
- evidência de persistência real validada por teste;
- evidência de reimportação sem duplicidade;
- evidência de consulta API sobre dado real;
- evidência de proteção do `document` nos testes relevantes.

---

## 10. Riscos e observações

- Se essa camada virar cópia lenta do unit test, foi mal feita.
- Se depender de ambiente manual, vai quebrar na hora errada.
- Se não validar idempotência e consulta real, não está cobrindo o coração do risco.
- Se tentar cobrir tudo em integração, a suíte vai virar lixo lento.
- Se ainda houver persistência fake em algum ponto, os testes podem passar mentindo.

---

## 11. Definição de pronto

A User Story será considerada pronta quando:

- a suíte de integração crítica estiver implementada;
- os principais fluxos reais de colaboração estiverem protegidos;
- os testes rodarem de forma reproduzível;
- os cenários de maior risco estiverem cobertos sem redundância burra;
- a base estiver pronta para a próxima etapa de Swagger, readiness e fechamento final.

---

## 12. Observação final

Esta User Story existe para proteger o que realmente importa depois da infraestrutura funcional: a colaboração real entre sync, banco e API.

Se ela for bem feita, você ganha confiança real. Se for mal feita, vira só ruído caro.
