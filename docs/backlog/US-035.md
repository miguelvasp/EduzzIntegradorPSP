# US-035 — Implementar base do mock server dos PSPs

## 1. Identificação

- **ID:** US-035
- **Título:** Implementar base do mock server dos PSPs
- **Épico:** Qualidade, testes e mock
- **Tipo:** Testes / Integração / DevEx
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-001 — Estruturar o projeto base da aplicação; US-002 — Definir a arquitetura modular e convenções de desenvolvimento; US-003 — Implantar convenção de enums, constantes, event types e error codes; US-004 — Configurar gerenciamento de configuração e segredos; US-013 — Implementar cliente e adapter do Pagar.me; US-014 — Implementar cliente e adapter do Mercado Pago; US-015 — Implementar Strategy e Factory para resolução de PSP

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** implementar a base do mock server dos PSPs  
**para** disponibilizar um ambiente controlado e previsível para desenvolvimento e testes, sem depender de sandbox externo, credenciais reais ou disponibilidade dos provedores.

---

## 3. Contexto de negócio

O desafio depende da integração com Pagar.me e Mercado Pago, porém o uso de ambientes reais ou sandbox pode introduzir instabilidade, limitação operacional e dificuldade para reproduzir cenários específicos. O próprio enunciado recomenda o uso de mock server como alternativa válida para não travar a entrega.

A solução precisa demonstrar sincronização paginada, ingestão idempotente, tratamento de erros, validação de negócio e consulta consistente. Para isso, é necessário um ambiente determinístico que reproduza os contratos mínimos esperados dos PSPs e permita evolução controlada da massa de teste.

---

## 4. Objetivo

Implementar a base do mock server para os PSPs suportados, contemplando:

- estrutura executável isolada para uso em desenvolvimento e testes;
- rotas mínimas compatíveis com os contratos consumidos pela aplicação;
- datasets versionados e reproduzíveis;
- configuração simples por ambiente;
- base para evolução de cenários funcionais e cenários de falha.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- criação da aplicação de mock server;
- definição da estrutura de rotas para Pagar.me e Mercado Pago;
- implementação dos endpoints mínimos de listagem e detalhe;
- uso de datasets locais versionados;
- configuração por ambiente para apontar clientes HTTP da aplicação para o mock;
- resposta compatível com o contrato esperado pelo desafio.

### 5.2 Fora do escopo
Esta User Story não contempla:

- simulação completa das APIs reais dos PSPs;
- webhooks;
- autenticação fiel ao comportamento real dos provedores;
- simulação avançada de falhas;
- integração com Docker Compose;
- cenários complexos de dados inconsistentes ou falhas técnicas, que serão tratados em histórias próprias.

---

## 6. Descrição funcional

O mock server deverá expor, no mínimo, os seguintes endpoints:

### Pagar.me
- `GET /core/v5/orders`
- `GET /core/v5/orders/:id`

### Mercado Pago
- `GET /v1/payments/search`
- `GET /v1/payments/:id`

As respostas deverão seguir o modelo esperado pelo desafio, permitindo que a aplicação consuma o mock sem necessidade de adaptações específicas no domínio.

O mock deverá fornecer uma massa padrão válida para:
- transações de cartão de crédito;
- dados básicos de pagador;
- parcelamento coerente;
- detalhes consistentes com a listagem.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

O mock server deverá ser implementado como artefato isolado de infraestrutura de teste, com baixo acoplamento à aplicação principal. A troca entre PSP real e mock deverá ocorrer por configuração, e não por alteração de código de domínio.

### 7.2 Endpoints mínimos obrigatórios

#### Pagar.me
- `GET /core/v5/orders?page=1&size=20`
- `GET /core/v5/orders/:id`

#### Mercado Pago
- `GET /v1/payments/search?payment_type_id=credit_card&offset=0&limit=20`
- `GET /v1/payments/:id`

### 7.3 Estrutura de datasets

A massa de dados deverá ser mantida em arquivos versionados, com separação mínima entre:
- cenários padrão válidos;
- identificadores consultáveis por detalhe;
- dados suficientes para paginação básica.

### 7.4 Compatibilidade com o desafio

A estrutura mínima de resposta deverá contemplar, no mínimo:

#### Pagar.me
- `data`
- `paging.total`
- `paging.has_more`

#### Mercado Pago
- `results`
- `paging.total`
- `paging.limit`
- `paging.offset`

### 7.5 Determinismo

Dado o mesmo dataset e os mesmos parâmetros, o mock deverá retornar sempre a mesma resposta.

### 7.6 Observabilidade mínima

O mock deverá registrar, no mínimo:
- rota chamada;
- parâmetros relevantes;
- cenário/dataset ativo;
- status de resposta.

---

## 8. Critérios de aceite

### 8.1 Funcionais
- O mock server sobe localmente de forma independente.
- Os endpoints mínimos de Pagar.me e Mercado Pago respondem com JSON compatível com o contrato esperado.
- A aplicação consegue apontar para o mock por configuração.
- Os endpoints de detalhe retornam dados coerentes com os registros de listagem.
- Há massa padrão suficiente para desenvolvimento inicial dos adapters.

### 8.2 Técnicos
- O mock não exige alteração de código de domínio para ser utilizado.
- Os datasets são versionados e reproduzíveis.
- A estrutura do mock é simples de manter e expandir.
- Há logs mínimos para troubleshooting local.

---

## 9. Definition of Done

- Mock server criado e executável.
- Rotas mínimas de Pagar.me e Mercado Pago implementadas.
- Dataset padrão válido criado.
- Configuração para uso do mock documentada.
- Base pronta para evolução dos cenários funcionais e de falha.
