# US-045 — Integrar Docker Compose com bootstrap funcional da aplicação

## 1. Identificação

- **ID:** US-045
- **Título:** Integrar Docker Compose com bootstrap funcional da aplicação
- **Épico:** Operação local e ambiente de validação
- **Tipo:** DevEx / Infra / Operação
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-004 — Configurar gerenciamento de configuração e segredos; US-038 — Implementar mock server dos PSPs para desenvolvimento e teste; US-040 — Implementar infraestrutura real de persistência no SQL Server; US-041 — Implementar persistência real da sincronização; US-042 — Implementar mecanismo acionável de sincronização de verdade; US-043 — Implementar os endpoints REST obrigatórios com consulta real; US-044 — Fechar o contrato HTTP e tratamento global de erro

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** integrar o Docker Compose com um bootstrap funcional da aplicação  
**para** garantir que o ambiente local suba de forma previsível, com banco, app e mock server realmente operáveis, reduzindo setup manual e risco de falha na validação técnica.

---

## 3. Contexto de negócio

Subir container não significa entregar ambiente funcional.

Hoje esse é um ponto onde muita solução morre feio: o `docker-compose up` até levanta processo, mas a aplicação não está pronta, o banco não tem bootstrap confiável, a API não responde de verdade, a sync não consegue executar, ou a ordem de subida quebra tudo.

No desafio, isso é mortal. O avaliador precisa conseguir:

- subir o ambiente;
- ter banco utilizável;
- ter app configurada corretamente;
- ter mock server acessível;
- disparar sincronização;
- consultar a API;
- fazer isso sem ritual obscuro.

Se o Compose só “sobe serviço”, mas não entrega ambiente funcional, então falhou no que importa.

---

## 4. Objetivo

Integrar o Docker Compose com bootstrap funcional da aplicação, contemplando:

- subida coordenada dos serviços essenciais;
- configuração consistente entre app, banco e mock server;
- inicialização previsível do banco;
- readiness mínima do ambiente;
- base confiável para testes, demo e validação final.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- ajuste do `docker-compose` para refletir o estado real da solução;
- integração funcional entre app, SQL Server e mock server;
- bootstrap inicial do banco para uso local/validação;
- ordenação e healthchecks pragmáticos dos serviços;
- configuração via variáveis de ambiente compatível com o projeto;
- documentação operacional mínima de subida e uso.

### 5.2 Fora do escopo
Esta User Story não contempla:

- Kubernetes;
- deploy em produção;
- hardening enterprise de container;
- alta disponibilidade;
- pipeline CI/CD completo;
- tuning avançado de imagem ou performance de container.

---

## 6. Descrição funcional

Ao executar `docker compose up`, o ambiente deverá ficar realmente utilizável para o fluxo principal do projeto.

Isso significa que, após a subida:

- o SQL Server deve estar acessível e pronto;
- a aplicação deve conseguir conectar no banco;
- o mock server deve estar acessível;
- a aplicação deve carregar sua configuração corretamente;
- a sincronização deve poder ser disparada;
- os endpoints REST obrigatórios devem poder ser consultados.

Não basta container “verde”. O ambiente precisa estar funcional.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

O Compose deve refletir o desenho real do projeto, sem teatro.

A configuração precisa ser pragmática, previsível e simples de rodar. Não é para criar topologia bonita e inútil.

### 7.2 Serviços mínimos esperados

O ambiente Compose deverá incluir, no mínimo:

- **app**
- **db**
- **mock-server**

Se houver outro serviço realmente necessário ao funcionamento atual do projeto, ele pode ser incluído. Mas não adicione perfumaria.

### 7.3 Banco

O SQL Server deve iniciar com estratégia de bootstrap coerente com a solução, contemplando o que for necessário para o ambiente funcionar, como por exemplo:

- criação de schema/tabelas;
- execução de migrations, se o projeto usar migrations;
- seed técnico mínimo, se for realmente necessário;
- readiness para uso pela aplicação.

O importante é isto: a aplicação não pode depender de intervenção manual obscura para começar a funcionar.

### 7.4 Aplicação

A aplicação no container deve:

- iniciar usando configuração de ambiente válida;
- conectar no banco real;
- apontar para mock ou real por configuração;
- expor as rotas esperadas;
- falhar claramente quando alguma dependência crítica não estiver pronta.

### 7.5 Ordem de subida e healthchecks

A solução deve tratar o mínimo necessário de coordenação entre serviços, por exemplo com:

- `depends_on` quando fizer sentido;
- healthcheck pragmático;
- lógica de retry/readiness na aplicação quando apropriado.

Não confunda isso com resolver tudo na marra. O objetivo é reduzir corrida de inicialização boba.

### 7.6 Configuração

As variáveis de ambiente precisam estar fechadas o suficiente para que o projeto rode sem gambiarra local escondida.

Isso inclui, no mínimo:

- conexão do SQL Server;
- portas;
- modo mock/real;
- base URL do mock server;
- parâmetros essenciais do app.

### 7.7 Persistência de dados

Se for útil para ambiente local e validação, o banco pode usar volume persistente. Se o desenho preferir ambiente efêmero para testes, isso deve estar claro.

O que não pode acontecer é o ambiente se comportar de forma imprevisível sem explicação.

### 7.8 Compatibilidade com fluxo do projeto

O ambiente Compose precisa suportar o fluxo real:

1. subir ambiente;
2. verificar app pronta;
3. disparar sync;
4. consultar endpoints;
5. opcionalmente rodar testes relevantes.

Se algum desses passos exigir truque fora do Compose/documentação mínima, a US não está fechada.

---

## 8. Critérios de aceite

### 8.1 Critérios funcionais
- `docker compose up` sobe app, db e mock server.
- O banco fica utilizável pela aplicação.
- A aplicação sobe com configuração válida.
- O mock server fica acessível pela aplicação.
- A sincronização pode ser executada no ambiente Compose.
- Os endpoints REST obrigatórios respondem no ambiente Compose.

### 8.2 Critérios técnicos
- A solução não depende de ajuste manual obscuro pós-subida.
- Existe bootstrap coerente do banco.
- A ordem de subida não quebra o funcionamento normal.
- A configuração é reprodutível em outra máquina.
- Não há quebra das US anteriores já concluídas.

### 8.3 Critérios operacionais
- O ambiente local fica utilizável para demo e validação.
- Logs e falhas de inicialização são compreensíveis.
- O fluxo principal pode ser reproduzido com poucos passos.

---

## 9. Evidências esperadas

- `docker compose up` funcional com os serviços essenciais;
- evidência de app conectando ao banco;
- evidência de mock server acessível;
- evidência de sincronização executada no ambiente Compose;
- evidência de consulta aos endpoints após a subida;
- documentação curta com os comandos principais.

---

## 10. Riscos e observações

- Se a app subir antes do banco sem estratégia mínima de prontidão, vai quebrar intermitentemente.
- Se o banco não tiver bootstrap real, o Compose vai ser só cenário vazio.
- Se o mock server não estiver integrado por configuração, o fluxo local vai ficar frágil.
- Se o Compose depender de segredos locais não documentados, a validação externa falha.
- Se só o processo subir, mas a API não funcionar, a entrega continua incompleta.

---

## 11. Definição de pronto

A User Story será considerada pronta quando:

- o ambiente Compose subir de forma reproduzível;
- app, banco e mock server estiverem integrados funcionalmente;
- a aplicação puder sincronizar e servir consulta no ambiente levantado;
- a configuração necessária estiver clara e mínima;
- os testes relevantes passarem;
- o fluxo principal puder ser demonstrado sem improviso.

---

## 12. Observação final

Esta User Story fecha o abismo entre “tenho containers” e “tenho ambiente funcional”.

Se ela ficar mal feita, a demonstração quebra mesmo com código bom.
