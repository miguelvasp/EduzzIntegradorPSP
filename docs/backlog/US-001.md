# US-001 — Estruturar o projeto base da aplicação

## 1. Identificação

- **ID:** US-001
- **Título:** Estruturar o projeto base da aplicação
- **Épico:** Fundação e arquitetura
- **Tipo:** Arquitetura / Plataforma
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** Nenhuma

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** estruturar o projeto base da aplicação com runtime, organização inicial, configurações e bootstrap padronizado  
**para** permitir a evolução segura das integrações, sincronização, persistência, API e testes sobre uma fundação técnica consistente.

---

## 3. Contexto de negócio

A solução deverá centralizar transações de cartão de crédito de múltiplos PSPs, mantendo integridade, rastreabilidade, segurança e capacidade de consulta unificada. Para atender esse objetivo, a aplicação precisa nascer com uma base técnica compatível com o escopo do desafio, incluindo runtime em Node.js com TypeScript, banco relacional, Docker, testes obrigatórios e organização compatível com princípios de legibilidade, separação de responsabilidades e evolução incremental. fileciteturn16file0

A estrutura inicial do projeto é responsável por viabilizar os próximos blocos do desenvolvimento: modelo canônico, integração com PSPs, sincronização incremental, API REST, segurança, observabilidade e documentação. O custo de uma fundação inconsistente nesta fase compromete diretamente a capacidade de entregar corretude funcional, qualidade de código, arquitetura, segurança e testes, todos avaliados com peso relevante no desafio. fileciteturn16file1

---

## 4. Objetivo

Estabelecer a fundação técnica do sistema com:

- projeto Node.js + TypeScript inicializado;
- servidor HTTP base com Fastify;
- organização modular do código;
- configuração centralizada de ambiente;
- pipeline inicial de execução local;
- estrutura de testes;
- convenções de documentação técnica e backlog;
- preparação para Docker, banco relacional e futuras integrações.

---

## 5. Escopo

### 5.1 Dentro do escopo

Esta User Story contempla:

- inicialização do projeto Node.js com TypeScript;
- configuração do compilador TypeScript;
- definição da estrutura de diretórios base da aplicação;
- bootstrap do servidor Fastify;
- criação da camada de configuração da aplicação;
- criação da camada de bootstrap do container/composição;
- criação da estrutura inicial de módulos de negócio;
- criação da estrutura inicial de testes;
- criação dos scripts básicos de desenvolvimento e build;
- criação do arquivo inicial de documentação técnica do projeto;
- criação do `.env.example`;
- padronização inicial de lint, formatação e convenções de nomes;
- criação de endpoint técnico mínimo de verificação da aplicação.

### 5.2 Fora do escopo

Esta User Story não contempla:

- modelagem do banco de dados;
- implementação de integrações com PSPs;
- sincronização de transações;
- endpoints funcionais do domínio;
- cache;
- outbox;
- risk/anti-fraud;
- regras de reconciliação;
- documentação funcional detalhada da API;
- pipelines de CI/CD.

---

## 6. Descrição funcional

A aplicação deverá ser iniciada localmente em modo de desenvolvimento e deverá disponibilizar uma base operacional mínima para evolução do produto. Essa base inclui organização de código, bootstrap do servidor, configuração por ambiente e um ponto de saúde técnica para validação da execução.

A estrutura do projeto deverá ser legível para manutenção por time, permitindo a separação entre camadas e módulos desde o início. O projeto deverá suportar o crescimento do escopo sem exigir reorganização estrutural prematura.

---

## 7. Descrição técnica

### 7.1 Stack base desta US

- **Runtime:** Node.js
- **Linguagem:** TypeScript
- **Framework HTTP:** Fastify
- **Validação de configuração:** camada centralizada de configuração
- **Testes:** estrutura inicial com Vitest
- **Documentação:** Markdown no repositório
- **Banco-alvo do projeto:** SQL Server
- **Containerização:** preparada para Docker Compose em histórias subsequentes

### 7.2 Estrutura inicial esperada

A base do projeto deverá contemplar a seguinte organização:

```text
src/
  app/
    server/
    config/
    container/
    cli/
    workers/

  modules/
    transactions/
    sync/
    reconciliation/
    psp/
    outbox/
    risk/
    shared/

  tests/
    unit/
    integration/
    e2e/

docs/
  backlog/
```

### 7.3 Responsabilidades técnicas desta US

#### a) Bootstrap da aplicação

Criar o ponto de entrada da aplicação com:

- inicialização do servidor;
- carregamento de configuração;
- registro de plugins essenciais;
- tratamento controlado de falha de inicialização.

#### b) Configuração centralizada

Criar módulo de configuração responsável por:

- ler variáveis de ambiente;
- validar presença mínima das configurações;
- expor configuração tipada para o restante do sistema;
- impedir acesso disperso e inconsistente a `process.env`.

#### c) Estrutura modular

Preparar os diretórios dos módulos principais para permitir evolução por domínio/capacidade, evitando organização genérica baseada apenas em tipo técnico.

#### d) Base de testes

Preparar estrutura inicial para:

- testes unitários;
- testes de integração;
- testes end-to-end.

#### e) Convenções transversais

Preparar convenções para:

- nomes de pastas e arquivos;
- separação entre `app`, `modules`, `tests` e `docs`;
- documentação versionada em Markdown;
- scripts de execução e build.

### 7.4 Arquivos mínimos esperados

A US deverá resultar, no mínimo, na criação ou preparação dos seguintes arquivos e diretórios:

```text
package.json
tsconfig.json
.env.example
.gitignore
src/app/server/createServer.ts
src/app/config/env.ts
src/app/container/index.ts
src/app/server/index.ts
src/modules/
src/tests/
docs/backlog/
README.md
```

### 7.5 Scripts mínimos esperados

O projeto deverá disponibilizar scripts equivalentes a:

- execução em desenvolvimento;
- build da aplicação;
- execução da aplicação compilada;
- execução de testes;
- validação de lint/qualidade, quando aplicável.

## 10. Critérios de aceite

### 10.1 Critérios funcionais

- O projeto deve iniciar sem erro em ambiente local com comando de desenvolvimento.
- A aplicação deve expor um endpoint técnico de verificação de disponibilidade.
- A estrutura de diretórios deve refletir a arquitetura modular definida.
- O carregamento de configuração deve ocorrer por módulo centralizado.
- A aplicação não deve depender de acesso direto e disperso a variáveis de ambiente em múltiplos pontos do código.
- Deve existir estrutura inicial de testes separada por tipo.
- Deve existir documentação técnica inicial no repositório.

### 10.2 Critérios técnicos

- O projeto deve utilizar TypeScript configurado e funcional.
- O servidor HTTP deve estar baseado em Fastify.
- A estrutura deve permitir evolução para integrações com PSPs, sincronização, API, outbox, cache e risk sem reorganização estrutural imediata.
- O código inicial deve manter separação clara entre bootstrap, módulos e documentação.
- O projeto não deve conter segredos hardcoded.
- O projeto deve estar preparado para containerização em histórias subsequentes.

### 10.3 Critérios de qualidade

- O repositório deve apresentar organização legível.
- Os nomes de arquivos e diretórios devem seguir convenção consistente.
- O projeto deve possuir baixo acoplamento estrutural na fundação.
- A base criada deve ser suficiente para permitir o início das próximas histórias sem retrabalho estrutural.

---

## 11. Mapeamento de testes

### 11.1 Testes unitários

- Validar carregamento de configuração obrigatória.
- Validar comportamento de falha quando configuração mínima estiver ausente.
- Validar criação do servidor sem dependências funcionais do domínio.

### 11.2 Testes de integração

- Validar subida da aplicação com configuração válida.
- Validar resposta do endpoint técnico de saúde.
- Validar bootstrap do servidor e registro da configuração.

### 11.3 Testes end-to-end

- Executar a aplicação em modo local.
- Realizar chamada HTTP ao endpoint técnico.
- Confirmar resposta compatível com aplicação operacional.

### 11.4 Evidências esperadas

- teste automatizado executável por script do projeto;
- aplicação iniciando localmente;
- endpoint técnico acessível;
- documentação técnica presente no repositório.

## 13. Riscos

- inicialização da aplicação sem convenção clara de estrutura, gerando retrabalho nas próximas histórias;
- acoplamento prematuro entre bootstrap e domínio;
- configuração dispersa e sem validação, aumentando risco operacional;
- ausência de base de testes desde o início, reduzindo confiabilidade da evolução;
- documentação tardia, dificultando acompanhamento técnico do backlog.

---

## 14. Definição de pronto

A User Story será considerada concluída quando:

- o projeto estiver inicializado e executável;
- a estrutura modular base estiver criada;
- a configuração centralizada estiver implementada;
- a base de testes estiver preparada;
- a documentação inicial estiver criada no repositório;
- os critérios de aceite estiverem atendidos;
- houver evidência executável da subida da aplicação e do endpoint técnico.
