# US-047 — Publicar Swagger/OpenAPI em /docs

## 1. Identificação

- **ID:** US-047
- **Título:** Publicar Swagger/OpenAPI em `/docs`
- **Épico:** API de consulta e experiência de validação
- **Tipo:** Backend / Documentação de API / Governança
- **Prioridade:** Média
- **Status inicial:** Backlog
- **Dependências:** US-043 — Implementar os endpoints REST obrigatórios com consulta real; US-044 — Fechar o contrato HTTP e tratamento global de erro; US-046 — Implementar testes de integração críticos

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** publicar a documentação Swagger/OpenAPI da API em `/docs`  
**para** permitir navegação clara do contrato, facilitar validação técnica e atender um diferencial relevante do desafio sem inventar documentação desconectada da implementação.

---

## 3. Contexto de negócio

Swagger aqui não é maquiagem. Mas também não é prioridade antes da API existir de verdade.

O edital trata a disponibilização de documentação Swagger/OpenAPI em `/docs` como diferencial valorizado, não como requisito obrigatório central. Então a ordem correta é simples: primeiro API real, depois contrato fechado, depois documentação navegável. Fazer isso antes é produzir mentira bonita. fileciteturn0file3L2-L4

Depois das US anteriores, a API já deve estar funcional e com contrato estável. A partir daí, publicar OpenAPI em `/docs` passa a ter valor real porque:

- reduz ambiguidade para avaliador;
- acelera teste manual;
- deixa explícitos os endpoints obrigatórios;
- documenta respostas de sucesso, paginação e erro;
- reforça a governança do contrato HTTP.

Sem isso, o avaliador fica dependente de README, código e tentativa e erro.

---

## 4. Objetivo

Publicar Swagger/OpenAPI em `/docs`, contemplando:

- especificação dos endpoints obrigatórios;
- documentação dos parâmetros principais;
- documentação de paginação;
- documentação do contrato de erro;
- UI navegável e utilizável no ambiente local.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- geração ou manutenção da especificação OpenAPI da API;
- publicação da UI Swagger em `/docs`;
- documentação dos 5 endpoints obrigatórios;
- documentação dos contratos de sucesso, paginação e erro;
- documentação básica dos parâmetros e respostas relevantes;
- alinhamento da documentação com a implementação real.

### 5.2 Fora do escopo
Esta User Story não contempla:

- portal externo de desenvolvedores;
- geração de SDK;
- versionamento complexo multi-API;
- documentação de endpoints internos não expostos;
- documentação manual extensa fora do escopo da API;
- inventar exemplos incompatíveis com o contrato real.

---

## 6. Descrição funcional

A aplicação deverá disponibilizar uma página navegável em:

- `GET /docs`

Nela, o avaliador ou desenvolvedor deverá conseguir ver, no mínimo:

- `GET /transactions`
- `GET /transactions/:id`
- `GET /transactions/:id/installments`
- `GET /installments/:id`
- `GET /transactions/:id/payer`

A documentação deverá refletir o comportamento real da API, incluindo:

- parâmetros aceitos;
- paginação;
- estruturas de resposta;
- erros comuns;
- limitações de segurança relevantes.

O objetivo não é enfeitar. O objetivo é deixar o contrato legível e testável.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

A documentação OpenAPI deve ser derivada ou mantida em total alinhamento com a implementação real.

Nada de escrever schema “bonito” que a API não cumpre.

### 7.2 Conteúdo mínimo da especificação

A especificação deve cobrir, no mínimo:

- rotas obrigatórias;
- parâmetros de rota;
- parâmetros de query relevantes;
- schemas de resposta de sucesso;
- schema paginado;
- schemas de erro relevantes;
- status HTTP principais.

### 7.3 Contratos obrigatórios a documentar

#### 7.3.1 Listagem de transações
Documentar:

- paginação;
- filtros disponíveis;
- shape da resposta;
- status principais.

#### 7.3.2 Detalhe de transação
Documentar:

- identificador esperado;
- retorno de sucesso;
- 404 quando aplicável.

#### 7.3.3 Parcelas da transação
Documentar:

- parâmetro de transação;
- retorno de sucesso;
- comportamento para transação inexistente, conforme o contrato real.

#### 7.3.4 Detalhe de parcela
Documentar:

- identificador da parcela;
- resposta esperada;
- erro de não encontrado.

#### 7.3.5 Pagador da transação
Documentar:

- contrato seguro de retorno;
- ausência de `document` puro;
- respostas principais.

### 7.4 Segurança do contrato

A documentação não pode sugerir comportamento inseguro.

Logo:

- não documentar `document` puro como campo de resposta;
- não inventar exemplos com CPF/CNPJ real;
- não documentar stack trace como resposta de erro;
- não expor detalhe interno desnecessário.

### 7.5 Integração com contrato real

Se o contrato HTTP mudou na US-044, o Swagger deve seguir exatamente esse contrato.

Se divergir, a documentação está errada. Não é discussão.

### 7.6 Ambiente local

A UI em `/docs` deve funcionar no ambiente local real da aplicação e, quando aplicável, no ambiente Docker Compose.

### 7.7 Manutenção

A abordagem escolhida deve ser pragmática. O ideal é reduzir a chance de drift entre código e especificação.

---

## 8. Critérios de aceite

### 8.1 Critérios funcionais
- A aplicação publica Swagger/OpenAPI em `/docs`.
- Os 5 endpoints obrigatórios estão documentados.
- A documentação permite navegação e entendimento do contrato.
- Paginação e erros relevantes estão documentados.
- O endpoint de pagador reflete política segura de exposição.

### 8.2 Critérios técnicos
- A especificação está alinhada ao comportamento real da API.
- Não há contrato inventado ou divergente da implementação.
- A UI funciona no ambiente local.
- A base fica pronta para demonstração e validação técnica.
- Não há quebra das US anteriores já concluídas.

### 8.3 Critérios de segurança
- A documentação não expõe `document` em texto puro.
- A documentação não usa exemplos com dado sensível real.
- A documentação não encoraja respostas inseguras.

---

## 9. Evidências esperadas

- acesso real a `/docs`;
- visualização dos 5 endpoints obrigatórios;
- evidência de schemas de sucesso, paginação e erro;
- evidência de alinhamento entre chamada real da API e documentação exibida.

---

## 10. Riscos e observações

- Se o Swagger for publicado antes da API estar estável, ele vai mentir.
- Se a documentação divergir do contrato HTTP real, o avaliador percebe.
- Se o endpoint de pagador expuser ou documentar dado sensível puro, você cria um problema desnecessário.
- Se `/docs` não funcionar no ambiente local, perdeu utilidade prática.

---

## 11. Definição de pronto

A User Story será considerada pronta quando:

- `/docs` estiver publicado e funcionando;
- os endpoints obrigatórios estiverem documentados corretamente;
- os contratos de sucesso, paginação e erro estiverem refletidos na especificação;
- a documentação estiver alinhada ao comportamento real da API;
- a solução estiver pronta para validação manual e demonstração.

---

## 12. Observação final

Esta User Story só faz sentido agora porque a API já deve existir de verdade.

Antes disso, Swagger seria só fachada. Agora ele vira reforço real de entrega.
