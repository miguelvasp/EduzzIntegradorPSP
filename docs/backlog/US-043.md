# US-043 — Implementar os endpoints REST obrigatórios com consulta real

## 1. Identificação

- **ID:** US-043
- **Título:** Implementar os endpoints REST obrigatórios com consulta real
- **Épico:** API de consulta e exposição funcional
- **Tipo:** Backend / API REST / Consulta
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-023 — Implementar endpoint de listagem paginada de transações; US-024 — Implementar endpoint de detalhe de transação; US-025 — Implementar endpoint de listagem de parcelas da transação; US-026 — Implementar endpoint de detalhe de parcela; US-027 — Implementar endpoint de consulta de pagador da transação; US-028 — Implementar contrato padronizado de resposta e paginação; US-029 — Implementar estratégia de cache para endpoints de consulta; US-030 — Implementar política de invalidação de cache baseada em atualização de transações; US-040 — Implementar infraestrutura real de persistência no SQL Server; US-041 — Implementar persistência real da sincronização; US-042 — Implementar mecanismo acionável de sincronização de verdade

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** implementar os endpoints REST obrigatórios consultando dados reais do banco  
**para** disponibilizar a interface unificada exigida pelo desafio, permitindo auditoria e consulta consistente das transações consolidadas após a sincronização.

---

## 3. Contexto de negócio

O desafio não pede uma arquitetura bonita isolada. Ele pede uma API útil.

Hoje esse é um dos maiores buracos do projeto: a aplicação ainda não entrega os endpoints obrigatórios funcionando de verdade sobre persistência real. Sem isso, não existe interface unificada para consulta, e o coração funcional do desafio continua faltando.

A API deve permitir que avaliadores e consumidores internos consigam:

- listar transações com paginação e filtros;
- consultar uma transação específica;
- consultar parcelas vinculadas a uma transação;
- consultar detalhe de uma parcela;
- consultar o pagador vinculado à transação;
- receber contratos HTTP consistentes e seguros.

Sem esses endpoints, o sistema não fecha o requisito central de auditoria e consulta.

---

## 4. Objetivo

Implementar os endpoints REST obrigatórios com consulta real, contemplando:

- leitura real no SQL Server;
- contrato HTTP consistente para sucesso e erro;
- paginação e filtros mínimos necessários;
- proteção de dados sensíveis;
- base concreta para Swagger, E2E, demonstração e entrega final.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- implementação dos endpoints REST obrigatórios do desafio;
- consulta real a partir das tabelas persistidas pela sincronização;
- paginação e filtros aplicáveis à listagem de transações;
- serialização segura dos dados expostos;
- contrato HTTP alinhado ao padrão do projeto;
- integração com cache existente, quando aplicável e já viável;
- compatibilidade com ambiente local e Docker Compose.

### 5.2 Fora do escopo
Esta User Story não contempla:

- criação de endpoints extras não exigidos;
- portal administrativo;
- mutações completas de escrita via API;
- autenticação enterprise completa se não fizer parte do desenho atual;
- filtros avançados além do necessário ao desafio;
- versionamento complexo da API.

---

## 6. Descrição funcional

A aplicação deverá disponibilizar, no mínimo, os endpoints REST exigidos pelo desafio, com leitura real sobre o banco:

- `GET /transactions`
- `GET /transactions/:id`
- `GET /transactions/:id/installments`
- `GET /installments/:id`
- `GET /transactions/:id/payer`

Esses endpoints deverão permitir que, após executar a sincronização:

- as transações consolidadas possam ser listadas;
- uma transação específica possa ser consultada;
- as parcelas vinculadas sejam recuperadas;
- uma parcela específica possa ser detalhada;
- o pagador vinculado seja retornado de forma segura.

A API deve refletir o estado real persistido pela sincronização, não dados fake em memória.

---

## 7. Descrição técnica

### 7.1 Diretriz técnica

A implementação deve usar consultas reais do banco e respeitar a separação de responsabilidades já desenhada no projeto.

Nada de controller fazendo regra de negócio ou SQL improvisado onde não deveria.

A solução deve conectar corretamente:

- rotas/controllers;
- schemas/validators de entrada;
- use cases ou query handlers;
- query repositories concretos;
- contrato HTTP padronizado;
- serialização segura de saída.

### 7.2 Endpoints obrigatórios

#### 7.2.1 Listagem paginada de transações
`GET /transactions`

Deve permitir, no mínimo, paginação e filtros compatíveis com o desafio e com o que já estiver no projeto, como por exemplo:

- `page`
- `pageSize`
- `psp`
- `status`
- intervalo de datas, quando já fizer parte do desenho atual

A resposta deve ser paginada e consistente.

#### 7.2.2 Detalhe de transação
`GET /transactions/:id`

Deve retornar os dados consolidados da transação correspondente ao identificador interno exposto pela API.

#### 7.2.3 Parcelas da transação
`GET /transactions/:id/installments`

Deve retornar as parcelas vinculadas à transação informada.

#### 7.2.4 Detalhe de parcela
`GET /installments/:id`

Deve retornar os dados da parcela específica.

#### 7.2.5 Pagador da transação
`GET /transactions/:id/payer`

Deve retornar os dados do pagador associados à transação, respeitando a política de segurança do projeto.

### 7.3 Regras obrigatórias de segurança

A API deve obedecer estas regras sem desculpa:

- nunca retornar `document` em texto puro;
- nunca retornar `documentHash` como se fosse o documento original;
- nunca expor stack trace bruto em erro;
- nunca vazar estrutura interna desnecessária.

Se o pagador precisar expor informação relacionada ao documento, isso deve seguir a política segura do projeto. O valor puro não pode sair.

### 7.4 Contratos HTTP

A API deve manter consistência de resposta para:

- sucesso;
- paginação;
- recurso não encontrado;
- erro de validação;
- erro inesperado.

Se já existe contrato padrão no projeto, ele deve ser reaproveitado. Não crie outro padrão por preguiça.

### 7.5 Paginação

A listagem de transações deve ter metadados de paginação confiáveis e coerentes com a consulta real.

Não vale paginação decorativa desligada do total real.

### 7.6 Query layer

A camada de consulta deve ser otimizada de forma pragmática:

- repositórios/query services concretos;
- joins ou queries compatíveis com o modelo real;
- sem puxar dado inútil;
- sem acoplamento burro entre API e persistência;
- sem cache fake.

### 7.7 Cache

Se a estratégia de cache já estiver pronta e utilizável sem gambiarras, integrá-la à consulta.

Se ainda não estiver madura, não inventar camada quebrada só para dizer que tem cache. O obrigatório aqui é a consulta real correta.

### 7.8 Validação de entrada

Parâmetros de rota e query string devem ser validados.

Não aceite payload inválido ou paginação absurda silenciosamente.

---

## 8. Critérios de aceite

### 8.1 Critérios funcionais
- Os 5 endpoints obrigatórios estão disponíveis e funcionando.
- A listagem de transações consulta dados reais persistidos.
- O detalhe de transação retorna o recurso correto.
- As parcelas vinculadas à transação são retornadas corretamente.
- O detalhe de parcela funciona corretamente.
- O endpoint de pagador retorna dados seguros e coerentes.
- A paginação funciona de verdade.
- Filtros implementados retornam resultado consistente.

### 8.2 Critérios técnicos
- Os endpoints usam query layer concreta ligada ao SQL Server.
- Não há uso de dados fake em memória para responder consulta.
- O contrato HTTP é consistente com o padrão do projeto.
- Não há quebra das US anteriores já concluídas.
- A solução permanece compatível com Docker/local.
- A implementação prepara o terreno para Swagger e E2E.

### 8.3 Critérios de segurança
- `document` nunca é exposto em texto puro.
- Erros não expõem detalhes internos indevidos.
- A serialização de resposta não vaza campos sensíveis desnecessários.

---

## 9. Evidências esperadas

- chamada real aos 5 endpoints obrigatórios;
- resposta paginada real da listagem de transações;
- resposta de detalhe de transação e de parcela;
- resposta segura do endpoint de pagador;
- evidência de leitura real após sincronização persistida;
- testes automatizados cobrindo os caminhos principais da API.

---

## 10. Riscos e observações

- Se a sync não persistir dados reais antes, esta US vira fachada.
- Se a API responder de memória, a entrega será fake.
- Se o pagador expuser documento puro, você viola o edital e mata a entrega.
- Se a paginação for decorativa, o avaliador percebe.
- Se o contrato HTTP variar entre endpoints, a API perde credibilidade.

---

## 11. Definição de pronto

A User Story será considerada pronta quando:

- os 5 endpoints obrigatórios estiverem implementados;
- todos consultarem o banco real;
- a resposta estiver segura e consistente;
- os testes relevantes passarem;
- a solução estiver utilizável em ambiente local e preparada para documentação Swagger e testes E2E.

---

## 12. Observação final

Esta User Story fecha o requisito funcional central do desafio: fornecer a interface unificada de consulta sobre dados consolidados.

Sem ela, o projeto continua sendo só pipeline interno e promessa.
