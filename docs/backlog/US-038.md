# US-038 — Integrar o mock server ao ambiente local, Docker Compose e fluxos de teste

## 1. Identificação

- **ID:** US-038
- **Título:** Integrar o mock server ao ambiente local, Docker Compose e fluxos de teste
- **Épico:** Qualidade, testes e mock
- **Tipo:** Infraestrutura / Testes / Integração
- **Prioridade:** Alta
- **Status inicial:** Backlog
- **Dependências:** US-035 — Implementar base do mock server dos PSPs; US-036 — Implementar cenários funcionais e de paginação no mock server dos PSPs; US-037 — Implementar cenários de falha técnica e payload inconsistente no mock server dos PSPs; US-042 — Empacotar a solução com Docker Compose; US-040 — Implementar testes de integração de persistência e sincronização; US-041 — Implementar testes end-to-end da API e da sincronização

---

## 2. História

**Como** time de engenharia responsável pelo agregador de transações multi-PSP  
**quero** integrar o mock server ao ambiente local, ao Docker Compose e aos fluxos de teste  
**para** executar a aplicação ponta a ponta com dependências controladas, sem depender de PSP real ou sandbox externa.

---

## 3. Contexto de negócio

O desafio permite o uso de mock server local como alternativa válida às APIs reais dos PSPs. Isso reduz risco de bloqueio por credenciais, aprovação de sandbox, instabilidade externa ou limitação operacional, e melhora a reprodutibilidade dos testes.

Entretanto, criar o mock isoladamente não basta. Ele precisa estar acoplado ao ambiente de execução e aos testes de forma simples, previsível e documentada.

---

## 4. Objetivo

Integrar o mock server ao ecossistema da solução, contemplando:

- execução conjunta com a aplicação e banco via Docker Compose;
- configuração por ambiente para alternar entre mock e PSP real;
- uso do mock nos testes de integração e end-to-end;
- documentação clara de como subir e operar o ambiente com mock.

---

## 5. Escopo

### 5.1 Dentro do escopo
Esta User Story contempla:

- inclusão do mock server no `docker-compose.yml`;
- configuração das variáveis de ambiente dos clients dos PSPs para apontar para o mock;
- definição do bootstrap de ambiente local com mock;
- wiring para que a aplicação rode sem alterar código de domínio;
- preparação do mock para suportar testes de integração e E2E;
- documentação operacional do uso do mock.

### 5.2 Fora do escopo
Esta User Story não contempla:

- criação de novos cenários de mock;
- alteração do domínio para acomodar mock;
- uso obrigatório de PSP real;
- pipelines avançadas de ambiente efêmero.

---

## 6. Descrição funcional

A solução deverá permitir que um desenvolvedor suba todo o ambiente com um único fluxo operacional, incluindo:

- aplicação principal;
- banco de dados;
- mock server dos PSPs;
- dependências adicionais previstas para o ambiente.

O ambiente deverá ser configurável para dois modos:

- **modo mock**, usado em desenvolvimento e testes locais;
- **modo real**, usado apenas se desejado para integração com PSP real.

A troca entre esses modos deverá ocorrer por configuração, sem necessidade de alterar regras de domínio, adapters ou casos de uso.

---

## 7. Descrição técnica

### 7.1 Integração com Docker Compose

O `docker-compose.yml` deverá incluir o mock server como serviço formal do ambiente, com:

- porta exposta;
- dependências mínimas;
- healthcheck, quando fizer sentido;
- integração com a aplicação principal via hostname interno de rede.

### 7.2 Configuração por ambiente

Os clients dos PSPs deverão ser configurados por variáveis de ambiente, incluindo base URLs específicas para:

- Pagar.me
- Mercado Pago

Essas URLs deverão apontar para o mock quando o ambiente local/teste estiver ativo.

### 7.3 Integração com testes

Os testes de integração e E2E deverão poder usar o mock server para validar:

- sincronização paginada;
- reimportação idempotente;
- atualização incremental;
- rejeição de payload inválido;
- tratamento de falhas técnicas simuladas.

### 7.4 Isolamento de responsabilidade

A aplicação de domínio não deverá conter ramificações do tipo “se mock então...”.  
A seleção entre mock e PSP real deverá ser inteiramente de infraestrutura/configuração.

### 7.5 Documentação operacional

A documentação deverá explicar, no mínimo:

- como subir o ambiente com mock;
- como alternar base URLs;
- como selecionar cenários do mock;
- como usar o mock nos fluxos de teste.

---

## 8. Critérios de aceite

### 8.1 Funcionais
- O mock server sobe junto com a aplicação no ambiente local.
- A aplicação consome o mock server sem alteração de código de domínio.
- Os testes de integração e E2E conseguem apontar para o mock.
- O ambiente local permite validar fluxo ponta a ponta com mock.

### 8.2 Técnicos
- O `docker-compose.yml` contempla o mock server.
- A troca entre mock e PSP real ocorre por configuração.
- A documentação operacional do mock está disponível.
- Não há acoplamento indevido do domínio ao ambiente de mock.

---

## 9. Definition of Done

- Mock server integrado ao `docker-compose.yml`.
- Variáveis de ambiente dos PSPs configuráveis para mock.
- Ambiente local executável com aplicação + banco + mock.
- Testes de integração/E2E aptos a usar mock.
- Documentação de uso do mock atualizada.
